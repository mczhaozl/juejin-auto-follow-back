# 掘金技术博客写作风格指南

## 核心原则

写技术文章要像讲故事，让读者产生共鸣，而不是干巴巴的技术文档。

## 文章结构模板

### 1. 开头：从"吃亏"说起（引发共鸣）

**目标**：让读者觉得"这不就是我吗？"

**写法**：
- 从一次具体的职场事件开始
- 描述遇到的问题和困境
- 强调痛点：沟通成本高、被甩锅、吃哑巴亏
- 情绪化表达：懵了、心里没底、心累

**示例开头**：
```
故事要从一次甩锅说起。

去年有段时间，我们有个项目打给客户内网部署。
客户那边反馈说「有个功能不对」，售前拉了个群，问我们：
现在他们内网跑的前端，到底是哪一版？

我懵了。
```

**关键词**：
- 甩锅、背锅、吃亏、冤枉
- 客户反馈、测试报 bug、领导质疑
- 沟通成本高、说不清、没证据
- 开会批斗、点名批评

### 2. 中间：尝试过的"笨办法"（制造对比）

**目标**：让读者觉得"我也试过，确实不行"

**写法**：
- 列举 2-3 个常见但不完美的解决方案
- 说明每个方案的缺点
- 用口语化的表达：要么...要么...、结果...、等于白写

**示例**：
```
一开始我试过在代码里手写版本号：
window.version = 'v1.2.3';  // 每次发版自己改

结果要么忘改，要么改错分支，上线后发现 version 还是上一版的，等于白写。
```

### 3. 转折：我的解决方案（亮出干货）

**目标**：让读者觉得"这个靠谱！"

**写法**：
- 简洁说明核心思路
- 强调自动化、不依赖人工
- 突出优势：不用记着改、不依赖某种工具

**示例**：
```
所以我自己写了一个插件。

查了一圈，没找到特别合胃口的，我就自己写了一个 XXX。
核心思路很简单：在打包时自动把 Git 信息塞进代码里。
```

### 4. 实现：技术细节（适度展开）

**目标**：让读者觉得"我也能做"

**写法**：
- 不要贴大段代码，只展示核心逻辑
- 用注释说明关键步骤
- 配图或示例效果图
- 强调简单、易用

**代码展示原则**：
- 核心代码 < 30 行
- 配置示例 < 10 行
- 效果截图 > 文字说明

### 5. 效果：解决了什么问题（回应开头）

**目标**：让读者觉得"确实有用"

**写法**：
- 对比使用前后的变化
- 用具体场景说明价值
- 强调省时间、避免甩锅、有证据

**示例**：
```
现在再有人说「你的包有问题」，我直接打开页面：
- 这是 main 分支
- Commit: abc123
- 打包时间：2024-01-15 10:30

谁也别想甩锅给我。
```

### 6. 结尾：分享 + 互动（引导点赞）

**目标**：让读者觉得"要支持一下"

**写法**：
- 分享 GitHub 仓库链接
- 说明开源、免费、欢迎使用
- 自然引导：觉得有用的话...
- 期望互动：点赞、收藏、关注

**示例**：
```
项目已开源，欢迎使用：
GitHub: https://github.com/xxx/xxx

如果这个工具帮到了你，欢迎点赞、收藏、关注三连！
有问题也欢迎在评论区交流 🎉
```

## 语言风格要求

### 1. 口语化表达

❌ 避免：在实际开发过程中，我们经常会遇到版本管理的问题
✅ 推荐：说实话，版本这事儿真的很烦

❌ 避免：该方案存在诸多弊端
✅ 推荐：这办法有几个坑

### 2. 情绪化词汇

多用：懵了、心累、绝了、离谱、没底、硬撑、吃亏

### 3. 短句为主

❌ 避免：由于客户环境我们无法访问，并且他们也不会提供构建日志，因此只能通过...
✅ 推荐：客户环境我们进不去。他们也不会给你看构建日志。最后只能靠...

### 4. 制造对比

- 之前 vs 现在
- 手动 vs 自动
- 没证据 vs 有证据
- 说不清 vs 一目了然

### 5. 使用 emoji

适度使用 emoji 增加亲和力：
- 😭 表示痛苦
- 💡 表示灵感
- ✅ 表示优点
- ❌ 表示缺点
- 🎉 表示成功

## 标题技巧

### 主标题格式

**模式一**：[工具名] + 一句话价值
- 例：version 一查，谁也别想甩锅给我

**模式二**：我踩坑了 + 问题描述
- 例：我踩坑了：线上跑的到底是哪一版代码？

**模式三**：[场景] + 解决方案
- 例：告别手动回关！用 GitHub Actions 打造自动化机器人

### 副标题格式

简短说明背景或结果：
- 测试说我给的包有问题，还开会批斗我
- 我写了个插件，下次他们再说有问题，我直接拿出证据

## 常见场景模板

### 场景一：被甩锅

```
故事要从一次甩锅说起。

[具体事件描述]

测试/客户/领导说：[问题]
我：[无法证明/说不清]
结果：[吃亏/背锅/开会批评]

我就想：[解决思路]
```

### 场景二：重复劳动

```
每次 [操作] 都要：
1. [步骤1]
2. [步骤2]
3. [步骤3]

如果 [情况]，这个过程能重复几十次...

作为程序员，怎么能容忍这种重复劳动？
```

### 场景三：工具不好用

```
一开始我试过 [方案A]，但 [缺点]
也试过 [方案B]，结果 [问题]
还试过 [方案C]，发现 [不足]

我就想：能不能 [理想方案]？
```

## 技术内容平衡

### 故事 vs 技术比例

- 开头故事：30%
- 技术实现：40%
- 效果展示：20%
- 互动引导：10%

### 代码展示原则

1. **少而精**：只展示核心代码
2. **有注释**：关键行加中文注释
3. **可运行**：确保代码能直接用
4. **有对比**：使用前 vs 使用后

### 配图建议

- 问题场景截图
- 效果对比图
- 使用示例图
- GitHub 仓库预览

## 互动引导技巧

### 自然引导

❌ 避免：请大家点赞关注
✅ 推荐：如果这个工具帮到了你，欢迎点赞支持

### 降低门槛

- 项目已开源，直接用
- 配置很简单，三行搞定
- 有问题评论区见

### 制造期待

- 后续会加入 [功能]
- 下一篇讲 [主题]
- 关注我，不迷路

## 检查清单

发布前检查：

- [ ] 开头有具体的职场故事
- [ ] 痛点描述让人有共鸣
- [ ] 列举了常见但不完美的方案
- [ ] 技术方案简洁易懂
- [ ] 有代码示例或效果图
- [ ] 强调了解决的价值
- [ ] 分享了 GitHub 链接
- [ ] 有互动引导
- [ ] 语言口语化、有情绪
- [ ] 标题吸引人
