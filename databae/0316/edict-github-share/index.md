# 发现了一个超硬核的 AI Agent 项目：用 1300 年前的三省六部制度来设计多 Agent 协作

> 当古人的政治智慧遇上现代 AI 框架，碰撞出了一个令人拍案叫绝的设计。

---

## 一、为什么我愿意花时间写这篇文章

作为一个关注 AI Agent 发展的人，我见过太多「换皮」的框架——换个名字，加几个 API 调用，就敢说是下一代多 Agent 协作方案。但最近在 GitHub 上发现的一个项目，让我眼前一亮。

这个项目叫 **Edict**，作者用了一个非常「反常识」的思路：用中国唐代的三省六部制度来设计 AI 多 Agent 协作架构。听起来很玄乎对吧？但看完架构设计后，我只能说——古人的分权制衡智慧，放到今天依然不过时。

## 二、大多数 Multi-Agent 框架的问题

在介绍 Edict 之前，我想先聊聊目前主流框架的通病。

CrewAI、AutoGen、LangChain Agents 这些框架，用过的人应该都有体会：它们的基本套路是「来，你们几个 AI 自己聊，聊完把结果给我」。听起来很简单，但实际用起来问题不少。

首先是**不可复现**。你让三个 Agent 讨论一个问题，第二次跑可能得到完全不同的结果，但你不知道中间发生了什么。

其次是**无法审计**。当结果出错时，你很难追溯是哪一步出了问题，哪个 Agent 说了什么不该说的话。

第三是**无法干预**。一旦流程跑起来，你就只能看着，很难中途调整方向。

这些问题，本质上是因为大多数框架**缺少制度性的约束**。Agent 之间的协作是松散的、随机的、不可预测的。

## 三、三省六部为什么适合 AI 协作

Edict 的作者做了一个很有趣的类比：现代 AI 框架的「多 Agent 自由讨论」，就像是没有宰相的朝堂——大家七嘴八舌，没有章法。

而三省六部制度，恰恰是一套**经过 1400 年验证的权力制衡机制**：

- **中书省**负责「出旨」——制定政策方案
- **门下省**负责「审核」——审议方案，不好的可以「封驳」
- **尚书省**负责「执行」——把政策分发到六部去办
- **六部**各司其职——户部管钱、礼部管文、兵部管工程、刑部管合规、工部管基建、吏部管人事

这套制度的精髓在于：**决策权、执行权、监督权分离**。任何政策都要经过「提出 → 审议 → 批准 → 执行 → 反馈」的完整流程，每个环节都有明确的责任人。

Edict 把这个思路搬到了 AI Agent 架构中。

## 四、Edict 的 12 Agent 架构

让我来拆解一下 Edict 的整体架构：

```
┌───────────────────────────────────┐
│          👑 皇上（你）              │
│     Feishu · Telegram · Signal     │
└─────────────────┬─────────────────┘
│ 下旨
┌─────────────────▼─────────────────┐
│          👑 太子 (taizi)            │
│    分拣：闲聊直接回 / 旨意建任务      │
└─────────────────┬─────────────────┘
│ 传旨
┌─────────────────▼─────────────────┐
│          📜 中书省 (zhongshu)       │
│       接旨 → 规划 → 拆解子任务       │
└─────────────────┬─────────────────┘
│ 提交审核
┌─────────────────▼─────────────────┐
│          🔍 门下省 (menxia)         │
│       审议方案 → 准奏 / 封驳 🚫      │
└─────────────────┬─────────────────┘
│ 准奏 ✅
┌─────────────────▼─────────────────┐
│          📮 尚书省 (shangshu)       │
│     派发任务 → 协调六部 → 汇总回奏    │
└───┬──────┬──────┬──────┬──────┬───┘
│      │      │      │      │
┌─────▼┐ ┌───▼───┐ ┌▼─────┐ ┌───▼─┐ ┌▼─────┐
│💰 户部│ │📝 礼部│ │⚔️ 兵部│ │⚖️ 刑部│ │🔧 工部│
│ 数据  │ │ 文档  │ │ 工程  │ │ 合规  │ │ 基建  │
└──────┘ └──────┘ └──────┘ └─────┘ └──────┘
┌──────┐
│📋 吏部│
│ 人事  │
└──────┘
```

这个架构有几个关键设计：

**1. 太子（消息分拣）**

用户下的「旨意」先经过太子分拣。闲聊性质的直接回复，只有正式的旨意才会被转成任务。这避免了大量无效信息进入后续流程。

**2. 三省（中书、门下、尚书）**

- **中书省**：接收旨意后进行规划，拆解成可执行的子任务
- **门下省**：审核方案，有权「封驳」不合理的规划
- **尚书省**：派发任务给六部执行，并汇总回奏

**3. 六部（并行执行）**

- **户部**：数据资源管理
- **礼部**：文档规范
- **兵部**：工程实现
- **刑部**：合规审计
- **工部**：基础设施
- **吏部**：人事调度

每个 Agent 都有独立的工作空间、技能配置和模型选择，职责边界非常清晰。

## 五、对比现有框架的优势

| 特性 | CrewAI | AutoGen | Edict |
|------|--------|---------|-------|
| 任务分发 | 简单轮询 | 动态协商 | 尚书省统一派发 |
| 结果审核 | 无 | 可选 | 门下省强制审核 |
| 状态追踪 | 基础 | 基础 | 完整 kanban 看板 |
| 权限控制 | 宽松 | 宽松 | 严格矩阵控制 |
| 干预能力 | 有限 | 有限 | 实时叫停/恢复 |

我最喜欢的是**门下省的封驳机制**。在传统框架中，方案一旦生成就很难撤回；而在 Edict 中，门下省可以拒绝不合理的规划，要求中书省重新考虑。

## 六、军机处看板：可视化监控

Edict 还提供了一个「军机处看板」，包含 10 个功能面板：

- **旨意看板**：kanban 视图展示所有任务，支持全文搜索和状态过滤
- **省部调度**：可视化各状态任务数量和部门分布
- **奏折阁**：已完成任务自动归档为「奏折」，保留完整时间线
- **旨库**：预设 9 个圣旨模板，一键下旨
- **官员总览**：Token 消耗排行、活跃度统计
- **天下要闻**：每日自动采集科技/财经资讯

这个看板让我想起了古代的「奏折」制度——所有决策都有记录可查，所有流程都可追溯。

## 七、一些技术细节

从 GitHub 仓库来看，Edict 的技术实现也很有特色：

- **dashboard.html**：单文件前端，零依赖，约 2500 行
- **server.py**：API 服务器，Python 标准库实现，约 2300 行
- **kanban_update.py**：看板 CLI，含旨意数据清洗和标题校验
- **skill_manager.py**：Skill 管理工具，支持远程/本地 Skills

每个 Agent 的配置都在 `agents/` 目录下，以 `SOUL.md` 形式定义人格模板。这种「人格即配置」的设计，让定制化变得非常简单。

## 八、我的思考

看完 Edict 的设计，我有几个感受：

**1. 跨学科思维的威力**

一个搞 AI 的人，去研究唐代的三省六部制度，并把两者结合起来——这种跨学科的思维方式非常难得。很多时候，答案不在你熟悉的领域，而在隔壁。

**2. 制度设计比技术实现更重要**

Edict 用到的技术（LLM API、Python、HTML）都很普通，但它对 Agent 协作流程的设计——任务怎么流转、谁有权审核、状态怎么转换——这些「制度」层面的设计，才是真正的护城河。

**3. 开源社区的创造力**

一个人能做出这样的项目，说明开源社区的创造力是无穷的。如果你对 AI Agent 感兴趣，一定要去 GitHub 上看看这个项目。

## 九、如何体验

Edict 的安装非常简单：

```bash
git clone https://github.com/cft0808/edict.git
cd edict
./install.sh
```

项目文档很详细，还有微信文章和截图可以参考。如果你正在做多 Agent 相关的项目，这个架构设计一定能给你一些启发。

## 写在最后

我在写这篇文章时一直在想一个问题：为什么是「三省六部」而不是西方的三权分立？

可能是因为三省六部更强调**流程的闭环**——从决策到执行到反馈，每个环节都有人负责。这种「可追溯、可干预、可审计」的特性，恰恰是当前 AI Agent 框架最需要的。

古人的智慧，经过 1300 年的时光，依然在 AI 时代闪闪发光。
