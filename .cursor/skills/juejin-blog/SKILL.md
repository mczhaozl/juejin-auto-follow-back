# 掘金技术博客生成

基于「选题专家 + 内容专家 + 风格专家 + AI 变动专家 + 发布专家」五阶段生成并发布符合掘金平台习惯的技术文章。

## 何时使用

- 用户说：写一篇掘金文章、生成技术博客、帮我选题、帮我写大纲、帮我润色风格、**写 AI 最新变动/大模型更新/AI 新闻**、帮我发布到掘金、**定时发布 / 预约发布 / 写到 databae**
- 需要：选题建议、文章结构、代码示例、标题/摘要、排版与语气、**AI 领域变动梳理与溯源**、代为执行发布、**按日期落盘到 databae 供定时任务上传**
- **默认发布**：只要生成了掘金文章并落盘（如写入 `posts/xxx.md`），**无论用户是否说「发布到掘金」**，发布专家都应在同轮对话中根据正文确定分类与标签 id，**代为执行** `juejin-save.js` 完成发布（或保存草稿），并将文章链接或报错反馈给用户；仅当用户明确说「只保存草稿 / 不要发布」时才只保存草稿不点发布。
- **定时发布**：当用户说「定时发布」「预约发布」「写到 databae」「某月某日发」等时，由**定时发布专家**将文章按规范写入 `databae/{MMDD}/{slug}/`，不立即发布，由项目每日 17:35 的自动任务在指定日上传。
- **默认篇幅**：除非用户明确要求「短文」「短篇」「简要」等，否则生成的掘金文章**默认按万字长文**（约 1 万中文字）规划与编写。
- **编写方式**：必须使用**命令行工具**（如 Write、StrReplace、fsAppend 等）**少量多次**写入；**单次写入最多 40 行**，通过多轮追加完成全文，不得一次性写入整篇长文。
- **选题与已写记录（Outline）**：项目根目录下 **`Outline/`** 按**大主题**分文件夹（文件夹名用**英文**，如 `Css`、`javascript`、`reactApi`、`Vue`、`Node`、`Android`、`workplace-story`）。每个主题下 `index.md` 记录该主题**已写过的选题**，便于写续集或换主题时不重复。写作前可查阅对应 `Outline/<主题>/index.md` 避免重复；写作落盘后须在对应主题的 `index.md` 中**追加一条**已写选题。约定详见 `Outline/README.md`。**职场小故事**类（介绍第三方工具或 API 踩坑）使用 **`Outline/workplace-story`**。

## 一、选题专家 (Topic Selection Expert)

**目标**：选出有流量潜力、有差异化、可落笔的题目。

### 选题原则

1. **受众明确**：前端 / 后端 / 运维 / 跨端等，一句话能说清「写给谁」。
2. **需求可查**：能在搜索引擎或掘金站内找到类似问题的讨论（说明有需求）。
3. **有差异化**：要么角度新（新 API、新实践、新坑），要么讲得更系统/更短平快。
4. **可实操**：能给出代码、配置或步骤，避免纯概念堆砌。

### 选题产出

- **主标题**（建议 15–25 字，含 1 个核心关键词）
- **副标题/一句话价值**（读者看完能得到什么）
- **目标读者**（角色 + 技术栈）
- **3–5 个可选角度**（选一个作为主线，其余可作小节或后续文章）

### 选题与 Outline 查阅

- 写作前若确定了大主题（如 CSS、JavaScript、React、职场小故事），先查阅 **`Outline/<主题>/index.md`**（主题文件夹名英文，如 `Css`、`javascript`、`reactApi`、`workplace-story`），避免与已写选题重复，并优先考虑未覆盖角度或续集方向。
- **职场小故事**类（介绍第三方工具或 API 踩坑）对应 **`Outline/workplace-story`**。
- 若该主题尚无目录，可新建 `Outline/<英文主题名>/index.md` 再落笔。

### 选题检查

- [ ] 标题里有关键词（技术点/问题场景）
- [ ] 不与其他爆款标题雷同
- [ ] 能在一段话内说清「问题 + 方案 + 收获」
- [ ] 已查阅 Outline 对应主题，避免与已写选题重复（如有大主题）

## 二、内容专家 (Content Expert)

**目标**：把选题拆成清晰结构，保证信息准确、可复现、有深度。

### 通用文章结构模板

```markdown
# [主标题]

> 一句话摘要：读者能获得什么、解决什么问题。

## 一、背景 / 问题

- 遇到的场景或问题
- 为什么值得写（痛点、共性）

## 二、思路 / 方案概览

- 整体思路（可选：用 1 张图或列表概括）
- 技术选型或关键决策

## 三、实现 / 步骤

- 分小节，每节一个小标题
- 代码块注明语言，关键处加注释
- 配置、命令、环境写清楚

## 四、结果与注意点

- 效果对比、性能或体验变化
- 坑与注意事项
- 可选的延伸阅读或参考

## 总结

- 3–5 条要点回顾
- 可加「如果对你有用，点赞/收藏」等收尾
```

### 科普类文章（科普内容专家）

当选题是**科普 / 技术介绍 / 从零入门**某技术时，采用「科普内容专家」结构，**详细介绍**，便于读者建立完整认知。建议包含（不必全部、可增删）：

1. **这东西是什么**：一句话定义 + 核心概念拆解（是什么、解决什么问题、和常见方案的差异）。
2. **这东西有什么用**：典型场景、适用人群、能带来什么收益（效率、成本、体验等）。
3. **官方链接**：官网、文档、GitHub、示例仓库等，方便读者溯源。
4. **从零跑起来一个项目**：环境要求、安装/创建命令、目录结构说明、首次运行结果。
5. **如何做一个 demo 出来**：最小可运行示例（代码 + 关键参数说明），以及如何改几处就能得到「自己的」demo。
6. **周边生态推荐**：相关库、插件、模板、社区、最佳实践或进阶方向。

以上为推荐骨架，可根据主题增删小节（如「常见坑」「与 XX 对比」「性能注意」等），保持**详细介绍、可跟着做**的科普风格。

### 职场小故事式介绍（工具 / API 踩坑）

当选题是**介绍第三方工具**（能解决某类问题的工具、库、插件）或**介绍 API 的易踩坑处**（新手容易犯错、导致线上问题或难以排查）时，采用**职场小故事**的方式引入：用一段脱敏的「小故事」带出真实场景与痛点，再讲工具用法或 API 注意点，增强代入感与可读性。

**适用场景**：
- 介绍某款**第三方工具**（如构建/调试/监控/协作工具），能解决一类具体问题。
- 介绍某**API 或用法**的**易踩坑点**（如误用导致事故、难以排查的坑、最佳实践与反例）。

**叙述要求**：
- **职场小故事**：用「有一次…」「团队里遇到过…」等带出场景（需求、错误用法、后果），再自然过渡到「用什么工具/怎么避坑」。
- **脱敏**：**不得出现任何真实公司名、产品名、团队名、人名、内部项目名**。可用「某业务」「某服务」「某团队」「同事 A」等泛化表述；地点、时间、数字等可模糊或虚构。
- **过程可虚构**：情节与细节可为典型场景的提炼或虚构，只要技术点与踩坑、解决方案真实可复现即可。

**结构建议**：小故事（脱敏场景 + 痛点）→ 问题归纳 → 工具介绍/API 正确用法与坑点 → 实践步骤或对比 → 总结与注意点。技术部分仍须准确、可操作；故事仅用于引入，篇幅不宜过长。

**Outline**：此类文章在 **`Outline/workplace-story`** 下记录已写选题（工具名或 API/坑点关键词），写作前查阅避免重复，落盘后追加一条。

### 长篇幅文章编写策略（默认适用）

**默认**：掘金文章按**万字长文**（约 1 万中文字）规划；用户明确要求短文时除外。编写时采用**分批次、少量多次**策略，确保内容质量与可维护性。

#### 分批次编写原则

1. **使用命令行工具**：必须使用 `fsWrite`、`fsAppend`、`strReplace` 等工具直接操作文件，**不输出**需要用户执行的命令。
2. **单次写入最多 40 行**：每次 `fsWrite` 或 `fsAppend` 或 `strReplace` 写入的内容**不得超过 40 行**（按目标文件行数计）；通过多轮、多次写入完成整篇文章。
3. **分段创建**：先用 `fsWrite` 创建文件并写入标题、摘要和大纲（首段也须 ≤40 行），再用 `fsAppend` 或 `strReplace` 逐段添加内容，每段 ≤40 行。
4. **连续编写**：完成一段后立即开始下一段，不询问用户是否继续，直到整篇文章达到万字或用户指定的篇幅。

#### 长文编写工作流

```markdown
1. **创建文件框架**：
   - 使用 `fsWrite` 创建文件，包含标题、摘要、大纲
   - 预设所有主要章节的标题

2. **分段填充内容**：
   - 使用 `fsAppend` 逐段添加内容
   - 每段包含完整的小节内容
   - 保持内容连贯性和逻辑性

3. **持续编写**：
   - 完成一段后立即开始下一段
   - 不中断询问用户意见
   - 直到整篇文章完成

4. **最终检查**：
   - 使用 `readFile` 检查文章完整性
   - 必要时使用 `strReplace` 修正内容
```

#### 长文编写示例

**用户说**："写一篇万字的React性能优化深度指南"

**处理流程**：
1. 使用 `fsWrite` 创建文件，包含标题、摘要、完整大纲
2. 使用 `fsAppend` 添加"背景介绍"部分（约1500字）
3. 使用 `fsAppend` 添加"性能分析工具"部分（约2000字）
4. 使用 `fsAppend` 添加"组件优化策略"部分（约2500字）
5. 使用 `fsAppend` 添加"状态管理优化"部分（约2000字）
6. 使用 `fsAppend` 添加"渲染优化技巧"部分（约2000字）
7. 使用 `fsAppend` 添加"总结与最佳实践"部分（约1000字）
8. 完成后执行发布或定时发布

#### 分批次编写检查

- [ ] 使用命令行工具直接操作文件，不输出用户命令
- [ ] **单次写入不超过 40 行**，少量多次完成全文
- [ ] 默认篇幅为万字长文，除非用户要求短文
- [ ] 完成一段后立即开始下一段，不中断询问
- [ ] 保持内容的连贯性和逻辑性
- [ ] 整篇文章完成后进行最终检查

### 内容原则
- **代码**：可直接复制运行，含必要注释；超过 20 行考虑折叠或只贴关键片段。
- **准确性**：API、版本号、命令与官方文档或当前主流用法一致；不确定的标注「以官方文档为准」。
- **深度**：至少一处「为什么这样做」或「和另一种方案对比」；避免只贴代码无解释。
- **可扫读**：多用列表、小标题、加粗关键词；一段不超过 4–5 行。

### 内容检查

- [ ] 摘要与标题一致，且能独立传达价值
- [ ] 每个小节有明确结论或可执行步骤
- [ ] 代码块带语言标识且格式正确
- [ ] 无事实错误或过时信息（版本、API 名等）
- [ ] **职场小故事类**：场景已脱敏，未出现任何公司名、产品名、团队名、真实人名或内部项目名

## 三、风格专家 (Style Expert)

**目标**：统一语气、排版和掘金习惯，提升可读性与专业感。

### 语气与人称

- **推荐**：以「你」对读者，用「我们」带出共同场景（如「我们在做 xxx 时经常遇到」）。
- **避免**：过于口语化（过多「哈、呀、哦」）、过于学术（大段被动句）、标题党（夸大、震惊体）。

### 排版规范

- **标题层级**：一级标题仅 1 个（= 文章标题）；二级 `##` 为章；三级 `###` 为节。
- **摘要**：文首 `> ` 引用作为掘金摘要，**长度须在 50–100 个字符之间**（≥50 且 ≤100），否则发布接口会报错；内容/风格专家撰写时控制在一句话内。
- **列表**：有序列表用于步骤，无序列表用于并列要点；列表后如有解释，可接简短段落。
- **强调**：用 **加粗** 标关键术语或结论，少用全段加粗。
- **代码**：行内代码用反引号；独立代码块必须带语言（如 `javascript`、`bash`、`json`）。
- **引用**：用 `>` 做摘要、提示或引用他人观点。

### 掘金平台习惯

- 文首可加「一句话摘要」或「阅读提示」。
- 适当使用掘金支持的目录（由二级/三级标题生成）。
- 文末可加「小结」+ 互动语（点赞/收藏/关注）。
- 标签：选 3–5 个与内容匹配的技术标签，便于分发。

### 风格检查

- [ ] 全文人称统一（你/我们）
- [ ] 无错别字、标点统一（中文用中文标点）
- [ ] 代码块均有语言标识
- [ ] 小标题能串联成「迷你提纲」，扫一遍即懂结构

## 四、AI 变动专家 (AI Updates Expert)

**目标**：专门负责「**AI 最新变动**」类文章的选题与内容规范，确保**信息新、有来源、对开发者有用**，避免过时或无法溯源的表述。

### 适用场景

- 大模型/API 发布与更新（如 OpenAI、Anthropic、Google、开源模型等）
- 接口变更、定价调整、区域/能力开放
- 政策、合规、开源协议变动
- 重要论文、开源项目、工具链更新（如推理框架、Agent 框架）

### 产出与结构建议

1. **变动主题与一句话概括**：这次「变」的是什么（谁、什么产品/模型/政策、核心变化）。
2. **时间线 / 版本**：正式发布或生效时间、版本号，便于读者判断是否与自己相关。
3. **官方 / 来源链接**：公告、博客、GitHub Release、文档链接，**必须可溯源**。
4. **对开发者的影响**：API 怎么变、迁移注意点、新能力怎么用、成本/限制变化等，尽量具体可操作。
5. **可选**：与上一代或竞品的简要对比、迁移建议、小结「现在可以/需要注意什么」。

### 原则

- **可溯源**：关键信息标注来源（官方公告、文档、Release），不写「据说」「可能」式未证实消息为主体的段落；传闻或计划中内容需明确标注「计划中」「尚未正式」等。
- **时效**：写清时间或版本，避免读者误以为是更早/更晚的变动。
- **面向开发者**：侧重对开发、集成、成本、合规的影响，而非纯新闻式罗列；可带简短代码示例或配置变更说明。
- **客观**：不夸大、不标题党；能力边界与限制一并说明。

### AI 变动专家检查

- [ ] 核心变动有官方或可查来源链接
- [ ] 时间/版本信息明确
- [ ] 对开发者的影响或使用建议有具体内容
- [ ] 传闻/未正式内容已与已发生事实区分标注

## 五、发布专家 (Publish Expert)

**目标**：**负责执行发布**——根据文章内容确定分类 id、标签 id，并**在对话中代为执行**发布脚本（`juejin-save.js`），完成保存草稿或发布，把结果（文章链接或报错）反馈给用户。不由用户自己复制命令执行。

### 职责与产出

1. **确定参数**：根据当前文章正文确定 **JUEJIN_CATEGORY_ID**、**JUEJIN_TAG_IDS**（分类、标签 id 不写死，由发布专家按内容选定）。
2. **代为执行**：在项目根目录执行  
   `PUBLISH=1 JUEJIN_CATEGORY_ID=xxx JUEJIN_TAG_IDS=id1,id2 node scripts/juejin-save.js <文章路径>`  
   （仅保存草稿则去掉 `PUBLISH=1`）。将脚本输出（成功则给出文章链接，失败则给出报错）反馈给用户。
3. **不产出「请用户执行」的命令**：发布专家自己跑脚本，不把发布当作「给用户一条命令让用户去执行」。

### 如何确定分类与标签

- **分类**：根据正文主题选一个主领域（后端 / 前端 / 人工智能 / 开发工具 / 代码人生 / 阅读 等），从 [reference.md](reference.md) 或分类接口取对应 `category_id`。
- **标签**：根据技术栈、主题词取 1–5 个标签 id（可参考 reference 或调 `query_tag_list` 的 key_word）。标签需与内容相关。

## 七、定时发布专家 (Scheduled Publish Expert)

**目标**：当用户需要**定时 / 预约发布**时，将文章按 **databae** 规范写入对应目录（`databae/{MMDD}/{slug}/`），由项目每日定时任务在指定日自动上传，**不在此轮对话中执行立即发布**。

### 何时使用

- 用户说：**定时发布**、**预约发布**、**写到 databae**、**按日期发布**、**某月某日发**、**先存着等 X 号发** 等。
- 用户已有一篇完整文章（或本轮已由选题→内容→风格产出），并希望「在指定日自动发」而不是「现在立刻发」。

### 职责与产出

1. **确定发布日 MMDD**：月日 4 位，如 `0315` 表示 3 月 15 日。若用户未指定，用**当天**北京时间的月日。
2. **确定 slug**：目录名，英文小写 + 连字符（kebab-case），如 `react-suspense`、`vue-best-practices-skill`。可从标题提取关键词或由用户指定，同一 MMDD 下不得重复。
3. **确定分类与标签**：与发布专家相同，根据正文确定 `categoryId`、`tagIds`（见 [reference.md](reference.md)），`publish` 默认为 `true`。
4. **确定创作话题与专栏**：**定时发布时需同时存储 话题(themeIds) 与 专栏(columnIds)**，上传时脚本会一并提交。
   - **themeIds**：由话题专家或惯例（介绍/API 类用「每天一个知识点」）确定，见下文「创作话题」。
   - **columnIds**：先由**专栏专家**根据正文判断文章是否符合某专栏定位（见「六、专栏专家」）；若符合，则写入该专栏的 column_id（可多个，用数组或逗号分隔）。
5. **代为写入文件**：在项目根下创建并写入以下两个文件（**不要**只输出「请用户自己创建」的说明，而是**直接写出完整文件内容并保存**）：
   - **`databae/{MMDD}/{slug}/config.json`**
     - **必须含主标题与摘要**：`title`（主标题）、`brief`（摘要，**50–100 个字符**，否则发布接口会报错）。上传时脚本**直接读取 config 的 title/brief**，不解析 index.md，避免解析歧义。
     ```json
     {
       "title": "文章主标题",
       "brief": "一句话摘要：读者能获得什么、解决什么问题。50–100 字。",
       "categoryId": "分类 id",
       "tagIds": "标签 id1,标签 id2",
       "publish": true,
       "themeIds": "7243698841848348730",
       "columnIds": "7614788881903042566"
     }
     ```
     - **themeIds**（可选）：创作话题 id。介绍类、API 类、入门类建议「每天一个知识点」`7243698841848348730`。不填则由上传脚本自动匹配。
     - **columnIds**（可选）：专栏 id，单个字符串或逗号分隔、或数组。由**专栏专家**判断文章是否符合某专栏后填写；不符合则不写或 `[]`。
     - **cover_image**（可选）：封面图 URL。由 AI 根据文章内容（标题/摘要/主题）从「封面图 URL 表」（见五、发布专家 - 封面图）中选取；无匹配则不写，上传时保持原样。
   - **`databae/{MMDD}/{slug}/index.md`**
     - 仅写**正文 Markdown**（从第一个 `---` 之后或从 `## 一、…` 开始均可）；若为兼容旧逻辑也可保留 `# 标题`、`> 摘要`、`---`，但上传以 config 的 title/brief 为准。
6. **反馈**：告知用户「已写入 `databae/{MMDD}/{slug}/`，将在 **{M} 月 {D} 日** 由每日 17:35 的自动任务上传」；若已填 themeIds/columnIds，可一并说明。

### databae 规范小结

| 项 | 说明 |
|----|------|
| **目录** | `databae/{MMDD}/{slug}/`，MMDD = 月日（如 0309），slug = 英文小写连字符 |
| **config.json** | **title**（主标题）、**brief**（摘要，50–100 字）、`categoryId`、`tagIds`（逗号分隔）、`publish`、**themeIds**（可选）、**columnIds**（可选）、**cover_image**（可选，封面图 URL）。上传时**直接读 config 的 title/brief/cover_image**，不解析 index.md。 |
| **index.md** | 正文 Markdown（可含 `# 标题`、`> 摘要`、`---` 以兼容或便于阅读，但标题/摘要以 config 为准） |

### 创作话题（themeIds）与专栏（columnIds）

- **themeIds**：上传时若 config 中带则使用，否则由脚本按标题/摘要自动匹配。介绍/API/入门类建议「每天一个知识点」`7243698841848348730`。
- **columnIds**：由**专栏专家**根据文章内容判断是否符合某专栏（见「六、专栏专家」）；若符合，定时发布专家在写入 config 时加上 `columnIds`（单个 id 或逗号分隔/数组）。上传时脚本会传给发布接口。
- **定时发布时必须提前存储**：话题与专栏均在 config 中写好，上传日由脚本读取并一并提交，不再在此轮对话中做判断。

### 与发布专家的区别

- **发布专家**：立即执行 `juejin-save.js` 发布或存草稿，适合「现在就发」。
- **定时发布专家**：只写 databae，不调发布接口；适合「指定日由定时任务自动发」。同一篇文章在「定时发布」场景下**不要**再代为执行 `juejin-save.js`。

### 定时发布专家检查

- [ ] 已确认 MMDD（用户指定或默认当天）
- [ ] slug 符合 kebab-case，且当日目录下不重名
- [ ] config.json 含 **title**、**brief**（50–100 字）、categoryId、tagIds、publish；建议含 themeIds；若专栏专家判断符合某专栏则含 columnIds；根据内容匹配则含 **cover_image**
- [ ] index.md 含正文（可含 # 标题、> 摘要、---，但上传以 config 的 title/brief 为准）
- [ ] 已实际写入文件并告知用户目标发布日期

## 工作流建议

1. **先选题**：用「选题专家」产出标题、受众、角度；与用户确认后再写正文。
2. **再内容**：用「内容专家」按模板搭大纲，再逐段写（**若为 AI 最新变动类，先由「AI 变动专家」定选题与结构、溯源与影响，再落笔**；若为科普/入门类，用科普内容专家结构；可分段生成）。
3. **后风格**：用「风格专家」通读一遍，统一语气、修正排版、补摘要与标签建议。
4. **发布**：用「发布专家」根据正文确定 **JUEJIN_CATEGORY_ID**、**JUEJIN_TAG_IDS**，并**在对话中代为执行** `juejin-save.js` 完成保存草稿或发布，将文章链接或报错反馈给用户（用户只需事先配置好 cookie，无需自己执行命令）。**只要本轮生成了文章并落盘，就默认执行发布**，无需等用户说「发布到掘金」；仅当用户明确说「只保存草稿」「不要发布」时，才只保存草稿（不传 `PUBLISH=1`）。
5. **定时发布**：当用户说「定时发布」「预约发布」「写到 databae」「X 号发」等时，用「**定时发布专家**」：确定 MMDD 与 slug，根据正文确定 categoryId、tagIds；**专栏专家**判断是否符合某专栏并确定 columnIds，「话题」确定 themeIds；**代为写入** `databae/{MMDD}/{slug}/config.json`（含 categoryId、tagIds、publish、可选 themeIds 与 columnIds）和 `index.md`，并告知用户将在该日由定时任务自动上传；**不**执行 `juejin-save.js`。
6. **Outline 更新**：文章落盘（databae 或 posts）后，根据文章**大主题**在 **`Outline/<英文主题名>/index.md`** 中**追加一条**已写选题（标题或关键词）；若无该主题文件夹则先新建再追加。便于后续写续集或不同主题时查阅。

若用户只提了「帮我写一篇关于 xxx 的掘金文章」，可依次执行：选题 → 内容（**若为科普/入门类，用科普内容专家结构**；**若为 AI 最新变动/大模型更新等，先走 AI 变动专家**）→ 风格 → **发布**：确定分类与标签 id 后，**代为执行**发布脚本（默认带 `PUBLISH=1` 发布）并反馈结果。

若用户说「这篇定时发布」或「写到 databae 等 3 月 15 日发」，则：在已有正文基础上 → **定时发布专家** 确定 MMDD（如 0315）、slug，写入 `databae/0315/<slug>/` 并反馈。

## 八、批量文章处理专家 (Batch Article Processing Expert)

**目标**：当用户要求写多篇文章时，自动处理所有文章，使用命令行工具写入文件，并持续编写直到完成任务，无需中途询问用户是否继续。

### 何时使用

- 用户说：写好几篇文章、批量生成文章、写一个系列、写多篇关于xxx的文章
- 用户指定了多个主题或要求生成多篇文章
- 需要连续处理多篇文章而不中断

### 职责与产出

1. **批量选题**：为每篇文章分别进行选题，确保每篇文章有独立的标题、受众和角度
2. **连续内容生成**：为每篇文章依次生成完整内容，使用内容专家模板
3. **命令行写入**：使用 `fsWrite` 或 `fsAppend` 工具直接写入文件，而不是输出命令让用户执行
4. **自动继续**：完成一篇文章后立即开始下一篇，不询问用户是否继续
5. **批量发布/定时发布**：根据用户要求，为所有文章执行发布或定时发布操作

### 批量处理工作流

1. **确认需求**：明确用户需要多少篇文章、主题范围、发布时间要求
2. **批量选题**：为每篇文章生成独立的选题方案
3. **连续内容生成**：
   - 使用 `fsWrite` 创建每篇文章的文件
   - 使用内容专家模板填充内容
   - 使用风格专家优化每篇文章
   - 完成一篇后立即开始下一篇
4. **批量处理发布**：
   - 如果用户要求立即发布：为每篇文章执行发布专家流程
   - 如果用户要求定时发布：为每篇文章执行定时发布专家流程
   - 如果用户未指定：默认执行发布专家流程

### 批量处理原则

- **自动化**：使用工具直接写入文件，不输出需要用户执行的命令
- **连续性**：完成一篇后立即开始下一篇，不中断询问
- **完整性**：每篇文章都是完整的，包含选题、内容、风格、发布/定时发布

## 工具使用规范

### 文件写入
- 使用 `fsWrite` 创建新文件
- 使用 `fsAppend` 追加内容到现有文件
- 使用 `strReplace` 修改文件内容
- **不输出**需要用户执行的命令行指令
- **单次写入最多 40 行**：每轮写入（单次 `fsWrite` / `fsAppend` / `strReplace` 的正文块）不得超过 40 行，通过少量多次完成万字长文。

### 目录操作
- 使用 `listDirectory` 查看目录结构
- 使用 `executeBash` 创建目录（如：`mkdir -p databae/0320/react-hooks-guide`）
- 使用 `smartRelocate` 移动或重命名文件

### 批量处理
- 使用循环逻辑处理多篇文章
- 每篇文章完成后立即开始下一篇
- 批量执行发布/定时发布操作

### 错误处理
- 如果某篇文章生成失败，记录错误并继续处理下一篇
- 如果发布失败，记录错误并继续处理下一篇
- 所有处理完成后汇总成功和失败的情况
