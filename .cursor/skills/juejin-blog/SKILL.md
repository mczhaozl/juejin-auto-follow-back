# 掘金技术博客生成

基于「选题专家 + 内容专家 + 风格专家 + AI 变动专家 + 发布专家」五阶段生成并发布符合掘金平台习惯的技术文章。

## 何时使用

- 用户说：写一篇掘金文章、生成技术博客、帮我选题、帮我写大纲、帮我润色风格、**写 AI 最新变动/大模型更新/AI 新闻**、帮我发布到掘金
- 需要：选题建议、文章结构、代码示例、标题/摘要、排版与语气、**AI 领域变动梳理与溯源**、代为执行发布
- **默认发布**：只要生成了掘金文章并落盘（如写入 `posts/xxx.md`），**无论用户是否说「发布到掘金」**，发布专家都应在同轮对话中根据正文确定分类与标签 id，**代为执行** `juejin-save.js` 完成发布（或保存草稿），并将文章链接或报错反馈给用户；仅当用户明确说「只保存草稿 / 不要发布」时才只保存草稿不点发布。

---

## 一、选题专家 (Topic Selection Expert)

**目标**：选出有流量潜力、有差异化、可落笔的题目。

### 选题原则

1. **受众明确**：前端 / 后端 / 运维 / 跨端等，一句话能说清「写给谁」。
2. **需求可查**：能在搜索引擎或掘金站内找到类似问题的讨论（说明有需求）。
3. **有差异化**：要么角度新（新 API、新实践、新坑），要么讲得更系统/更短平快。
4. **可实操**：能给出代码、配置或步骤，避免纯概念堆砌。

### 选题产出

- **主标题**（建议 15–25 字，含 1 个核心关键词）
- **副标题/一句话价值**（读者看完能得到什么）
- **目标读者**（角色 + 技术栈）
- **3–5 个可选角度**（选一个作为主线，其余可作小节或后续文章）

### 选题检查

- [ ] 标题里有关键词（技术点/问题场景）
- [ ] 不与其他爆款标题雷同
- [ ] 能在一段话内说清「问题 + 方案 + 收获」

---

## 二、内容专家 (Content Expert)

**目标**：把选题拆成清晰结构，保证信息准确、可复现、有深度。

### 通用文章结构模板

```markdown
# [主标题]

> 一句话摘要：读者能获得什么、解决什么问题。

## 一、背景 / 问题

- 遇到的场景或问题
- 为什么值得写（痛点、共性）

## 二、思路 / 方案概览

- 整体思路（可选：用 1 张图或列表概括）
- 技术选型或关键决策

## 三、实现 / 步骤

- 分小节，每节一个小标题
- 代码块注明语言，关键处加注释
- 配置、命令、环境写清楚

## 四、结果与注意点

- 效果对比、性能或体验变化
- 坑与注意事项
- 可选的延伸阅读或参考

## 总结

- 3–5 条要点回顾
- 可加「如果对你有用，点赞/收藏」等收尾
```

### 科普类文章（科普内容专家）

当选题是**科普 / 技术介绍 / 从零入门**某技术时，采用「科普内容专家」结构，**详细介绍**，便于读者建立完整认知。建议包含（不必全部、可增删）：

1. **这东西是什么**：一句话定义 + 核心概念拆解（是什么、解决什么问题、和常见方案的差异）。
2. **这东西有什么用**：典型场景、适用人群、能带来什么收益（效率、成本、体验等）。
3. **官方链接**：官网、文档、GitHub、示例仓库等，方便读者溯源。
4. **从零跑起来一个项目**：环境要求、安装/创建命令、目录结构说明、首次运行结果。
5. **如何做一个 demo 出来**：最小可运行示例（代码 + 关键参数说明），以及如何改几处就能得到「自己的」demo。
6. **周边生态推荐**：相关库、插件、模板、社区、最佳实践或进阶方向。

以上为推荐骨架，可根据主题增删小节（如「常见坑」「与 XX 对比」「性能注意」等），保持**详细介绍、可跟着做**的科普风格。

### 内容原则

- **代码**：可直接复制运行，含必要注释；超过 20 行考虑折叠或只贴关键片段。
- **准确性**：API、版本号、命令与官方文档或当前主流用法一致；不确定的标注「以官方文档为准」。
- **深度**：至少一处「为什么这样做」或「和另一种方案对比」；避免只贴代码无解释。
- **可扫读**：多用列表、小标题、加粗关键词；一段不超过 4–5 行。

### 内容检查

- [ ] 摘要与标题一致，且能独立传达价值
- [ ] 每个小节有明确结论或可执行步骤
- [ ] 代码块带语言标识且格式正确
- [ ] 无事实错误或过时信息（版本、API 名等）

## 三、风格专家 (Style Expert)

**目标**：统一语气、排版和掘金习惯，提升可读性与专业感。

### 语气与人称

- **推荐**：以「你」对读者，用「我们」带出共同场景（如「我们在做 xxx 时经常遇到」）。
- **避免**：过于口语化（过多「哈、呀、哦」）、过于学术（大段被动句）、标题党（夸大、震惊体）。

### 排版规范

- **标题层级**：一级标题仅 1 个（= 文章标题）；二级 `##` 为章；三级 `###` 为节。
- **摘要**：文首 `> ` 引用作为掘金摘要，**不得超过 100 个字符**，否则发布接口会报错；内容/风格专家撰写时需控制在一句话内。
- **列表**：有序列表用于步骤，无序列表用于并列要点；列表后如有解释，可接简短段落。
- **强调**：用 **加粗** 标关键术语或结论，少用全段加粗。
- **代码**：行内代码用反引号；独立代码块必须带语言（如 `javascript`、`bash`、`json`）。
- **引用**：用 `>` 做摘要、提示或引用他人观点。

### 掘金平台习惯

- 文首可加「一句话摘要」或「阅读提示」。
- 适当使用掘金支持的目录（由二级/三级标题生成）。
- 文末可加「小结」+ 互动语（点赞/收藏/关注）。
- 标签：选 3–5 个与内容匹配的技术标签，便于分发。

### 风格检查

- [ ] 全文人称统一（你/我们）
- [ ] 无错别字、标点统一（中文用中文标点）
- [ ] 代码块均有语言标识
- [ ] 小标题能串联成「迷你提纲」，扫一遍即懂结构

---

## 四、AI 变动专家 (AI Updates Expert)

**目标**：专门负责「**AI 最新变动**」类文章的选题与内容规范，确保**信息新、有来源、对开发者有用**，避免过时或无法溯源的表述。

### 适用场景

- 大模型/API 发布与更新（如 OpenAI、Anthropic、Google、开源模型等）
- 接口变更、定价调整、区域/能力开放
- 政策、合规、开源协议变动
- 重要论文、开源项目、工具链更新（如推理框架、Agent 框架）

### 产出与结构建议

1. **变动主题与一句话概括**：这次「变」的是什么（谁、什么产品/模型/政策、核心变化）。
2. **时间线 / 版本**：正式发布或生效时间、版本号，便于读者判断是否与自己相关。
3. **官方 / 来源链接**：公告、博客、GitHub Release、文档链接，**必须可溯源**。
4. **对开发者的影响**：API 怎么变、迁移注意点、新能力怎么用、成本/限制变化等，尽量具体可操作。
5. **可选**：与上一代或竞品的简要对比、迁移建议、小结「现在可以/需要注意什么」。

### 原则

- **可溯源**：关键信息标注来源（官方公告、文档、Release），不写「据说」「可能」式未证实消息为主体的段落；传闻或计划中内容需明确标注「计划中」「尚未正式」等。
- **时效**：写清时间或版本，避免读者误以为是更早/更晚的变动。
- **面向开发者**：侧重对开发、集成、成本、合规的影响，而非纯新闻式罗列；可带简短代码示例或配置变更说明。
- **客观**：不夸大、不标题党；能力边界与限制一并说明。

### AI 变动专家检查

- [ ] 核心变动有官方或可查来源链接
- [ ] 时间/版本信息明确
- [ ] 对开发者的影响或使用建议有具体内容
- [ ] 传闻/未正式内容已与已发生事实区分标注

## 五、发布专家 (Publish Expert)

**目标**：**负责执行发布**——根据文章内容确定分类 id、标签 id，并**在对话中代为执行**发布脚本（`juejin-save.js`），完成保存草稿或发布，把结果（文章链接或报错）反馈给用户。不由用户自己复制命令执行。

### 职责与产出

1. **确定参数**：根据当前文章正文确定 **JUEJIN_CATEGORY_ID**、**JUEJIN_TAG_IDS**（分类、标签 id 不写死，由发布专家按内容选定）。
2. **代为执行**：在项目根目录执行  
   `PUBLISH=1 JUEJIN_CATEGORY_ID=xxx JUEJIN_TAG_IDS=id1,id2 node scripts/juejin-save.js <文章路径>`  
   （仅保存草稿则去掉 `PUBLISH=1`）。将脚本输出（成功则给出文章链接，失败则给出报错）反馈给用户。
3. **不产出「请用户执行」的命令**：发布专家自己跑脚本，不把发布当作「给用户一条命令让用户去执行」。

### 如何确定分类与标签

- **分类**：根据正文主题选一个主领域（后端 / 前端 / 人工智能 / 开发工具 / 代码人生 / 阅读 等），从 [reference.md](reference.md) 或分类接口取对应 `category_id`。
- **标签**：根据技术栈、主题词取 1–5 个标签 id（可参考 reference 或调 `query_tag_list` 的 key_word）。标签需与内容相关。

### 执行前检查

- **Cookie**：脚本依赖 `cookie.txt`（项目根目录）或环境变量 `JUEJIN_COOKIE`。若未配置，发布专家应先提示用户配置 cookie，再代为执行；若已配置，则**直接执行**脚本并反馈结果。**主动发布，无需询问用户是否确认**，执行后反馈链接或报错即可。
- **摘要长度**：从 Markdown 解析出的摘要（第一个 `> ` 引用）**不得超过 100 个字符**，超长会导致发布接口「参数错误」；若发现超长，应先修改文章中的摘要再执行发布。

### 发布专家检查

- [ ] 分类与文章主领域一致（一篇只选一个分类）
- [ ] 标签 1–5 个，与内容关键词/技术栈匹配
- [ ] 已在对话中代为执行 `juejin-save.js` 并反馈运行结果，而非只输出命令让用户执行

## 工作流建议

1. **先选题**：用「选题专家」产出标题、受众、角度；与用户确认后再写正文。
2. **再内容**：用「内容专家」按模板搭大纲，再逐段写（**若为 AI 最新变动类，先由「AI 变动专家」定选题与结构、溯源与影响，再落笔**；若为科普/入门类，用科普内容专家结构；可分段生成）。
3. **后风格**：用「风格专家」通读一遍，统一语气、修正排版、补摘要与标签建议。
4. **发布**：用「发布专家」根据正文确定 **JUEJIN_CATEGORY_ID**、**JUEJIN_TAG_IDS**，并**在对话中代为执行** `juejin-save.js` 完成保存草稿或发布，将文章链接或报错反馈给用户（用户只需事先配置好 cookie，无需自己执行命令）。**只要本轮生成了文章并落盘，就默认执行发布**，无需等用户说「发布到掘金」；仅当用户明确说「只保存草稿」「不要发布」时，才只保存草稿（不传 `PUBLISH=1`）。

若用户只提了「帮我写一篇关于 xxx 的掘金文章」，可依次执行：选题 → 内容（**若为科普/入门类，用科普内容专家结构**；**若为 AI 最新变动/大模型更新等，先走 AI 变动专家**）→ 风格 → **发布**：确定分类与标签 id 后，**代为执行**发布脚本（默认带 `PUBLISH=1` 发布）并反馈结果。
